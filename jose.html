📚 ¿Qué es JWT?

JSON Web Token (JWT) es un estándar abierto (RFC 7519) que define un formato compacto y autocontenido para transmitir información entre partes como un objeto JSON.

🔨 Estructura de un JWT

Header . Payload . Signature

Header

Metadatos del token

{ "alg": "HS256", "typ": "JWT" }

Payload

Datos (claims)

{ "sub": "123", "role": "admin" }

Signature

Firma verificable

HMACSHA256(header.payload, secret)

🔒 Conceptos de Seguridad

⚠️

Riesgos Sin Autenticación

  • • Cualquier persona puede modificar datos
  • • No hay forma de identificar al usuario
  • • Ataques de suplantación de identidad
  • • Acceso no autorizado a recursos sensibles
✅

Beneficios de JWT

  • • Autenticación stateless (sin sesiones)
  • • Información verificable y confiable
  • • Expiración configurable
  • • Protección contra manipulación
💡

Buena Práctica: Secretos Fuertes

Usa secretos largos y aleatorios (mínimo 32 caracteres). En producción, nunca hardcodées secretos en el código. Usa variables de entorno (process.env.JWT_SECRET).

✍️ Paso 3: Utilitario JWT

Crea src/utils/jwt.js:

import { SignJWT } from 'jose';
import 'dotenv/config';

const secret = new TextEncoder().encode(process.env.JWT_SECRET);

export const generateToken = async (payload) => {
  return await new SignJWT(payload)
    .setProtectedHeader({ alg: 'HS256' })
    .setIssuedAt()
    .setExpirationTime('24h')
    .sign(secret);
};

🔑 SignJWT

Crea un token firmado

🛡️ setProtectedHeader

Define HS256 como algoritmo

⏰ setExpirationTime

Token expira en 24 horas

✍️ sign(secret)

Firma con clave secreta

🛡️ Paso 4: Middleware de Autenticación

Crea src/middlewares/auth.js:

import { jwtVerify } from 'jose';
import 'dotenv/config';

const secret = new TextEncoder().encode(process.env.JWT_SECRET);

export const authMiddleware = async (req, res, next) => {
  const authHeader = req.headers.authorization;

  if (!authHeader || !authHeader.startsWith('Bearer ')) {
    return res.status(401).json({ error: 'Token no proporcionado' });
  }

  const token = authHeader.split(' ')[1];

  try {
    const { payload } = await jwtVerify(token, secret);
    req.user = payload;
    next();
  } catch (error) {
    return res.status(401).json({ error: 'Token inválido o expirado' });
  }
};

🔍 Análisis del Middleware

1

Extrae el header Authorization

2

Verifica que tenga formato Bearer <token>

3

Valida la firma y expiración con jwtVerify()

4

Si es válido, guarda el payload en req.user

🔑 Paso 5: Endpoint de Login

Crea src/routes/auth.routes.js:

import { Router } from 'express';
import { users } from '../models/users.model.js';
import { generateToken } from '../utils/jwt.js';

const router = Router();

router.post('/login', async (req, res) => {
  const { email, password } = req.body;

  if (!email || !password) {
    return res.status(400).json({ error: 'Email y password son requeridos' });
  }

  const user = users.find((u) => u.email === email && u.password === password);

  if (!user) {
    return res.status(401).json({ error: 'Credenciales inválidas' });
  }

  const token = await generateToken({
    id: user.id,
    role: user.role
  });

  res.json({ token });
});

export default router;

💡 ¿Por qué solo id y role?

El payload del JWT es visible (solo está firmado, no encriptado). Por seguridad, solo incluimos lo mínimo necesario: id para identificar y role para permisos. Nombre y email se obtienen del servidor si es necesario.

🔒 Paso 6: Proteger Rutas

Aplica el middleware a rutas de modificación:

src/routes/users.routes.js

import { Router } from 'express';
import { authMiddleware } from '../middlewares/auth.js';
// ... import controllers

const router = Router();

router.get('/', getUsers);           // ✅ Público - cualquier persona puede ver
router.get('/:id', getUserById);     // ✅ Público
router.post('/', authMiddleware, createNewUser);    // 🔒 Protegido
router.put('/:id', authMiddleware, updateExistingUser);  // 🔒 Protegido
router.delete('/:id', authMiddleware, deleteExistingUser); // 🔒 Protegido

export default router;

✅ GET (Lectura)

Acceso libre, no requiere token

🔒 POST/PUT/DELETE (Escritura)

Requiere token válido en header

⚙️ Paso 7: Integración en app.js

Registra las rutas en src/app.js:

import express from "express";
import cors from "cors";
import morgan from "morgan";
import usersRoutes from "./routes/users.routes.js";
import postsRoutes from "./routes/post.routes.js";
import authRoutes from "./routes/auth.routes.js";
import { sanitizeMiddleware } from "./middlewares/sanitize.js";
import { errorHandler } from "./middlewares/errorHandler.js";

const app = express();

app.use(cors());
app.use(express.json());

if (process.env.NODE_ENV !== "production") {
  app.use(morgan("dev"));
}

app.use(sanitizeMiddleware);    // 1. Limpiar datos de entrada
app.use("/auth", authRoutes);  // 2. Rutas de autenticación
app.use("/users", usersRoutes); // 3. Rutas de usuarios
app.use("/posts", postsRoutes); // 4. Rutas de posts

app.use((req, res) => {
  res.status(404).json({ error: "Ruta no encontrada" });
});

app.use(errorHandler);        // 5. Manejo de errores global

export default app;
💡

Orden de Middlewares

El orden es crucial. Los middlewares se ejecutan en secuencia. Primero sanitizamos, luego autenticamos, después van las rutas.

📝 Conceptos Clave

🔐 Autenticación vs Autorización

  • Autenticación: ¿Quién eres?
  • Autorización: ¿Qué puedes hacer?

🕐 Stateless vs Stateful

  • JWT: Stateless - el token contiene todo
  • Sesiones: Stateful - el servidor guarda estado

⏰ Expiración de Tokens

  • Tokens de corta duración (15-60 min) para acceso
  • Refresh tokens para renew automático

🔄 Roles y Permisos

  • admin: Acceso total
  • user: Lectura y escritura propia
  • guest: Solo lectura

🔒 Payload Mínimo en JWT

  • El payload es visible (solo firmado, no encriptado)
  • Solo incluir id y role
  • Nunca incluir passwords o datos sensibles

🚫 Sanitización de Contraseñas

  • Nunca retornar passwords en respuestas API
  • Usar destructuring: const {'{ password, ...user }'}
  • Internamente se guardan, pero nunca se exponen
